# Стратегия продукта: сканер нарушений GDPR/ePrivacy для сайтов Латвии и ЕС

> Рабочее название: **Violation Scanner** / «Compliance Checker»
> Статус: прототип для проверки идеи. Документ описывает полное видение продукта — от MVP до «суперфич». Технический план MVP уже согласован и реализуется параллельно (см. `~/.claude/plans/rosy-chasing-petal.md`); этот документ — более широкая продуктовая стратегия и каталог возможностей.

---

## 1. Видение продукта (elevator pitch)

Пользователь вставляет ссылку на **любой сайт**. Сервис за 10–30 секунд открывает его в реальном браузере, ловит, что происходит *на самом деле* (какие куки и трекеры грузятся до согласия, есть ли баннер, доступна ли политика конфиденциальности, работает ли HTTPS), сопоставляет находки с базой правил «нарушение → статья закона → диапазон штрафа» и выдаёт понятный отчёт: **что не так, где именно на странице, и во сколько это может обойтись**, если не исправить.

**Ключевое отличие от рынка** (подтверждено исследованием): ни Cookiebot, ни CookieYes, ни Osano не показывают оценку штрафа. FineMyData оценивает штраф, но по анкете, а не по сканированию сайта — и никогда не показывает «где». Мы — единственные, кто соединяет все три элемента (что / где / сколько) в одном простом инструменте.

## 2. Целевая аудитория

**v1 (сейчас):** широкий охват — любой сайт, который вставит пользователь. Это подтверждено в основной ветке решений сессии.

**Приоритетный сегмент для UX-примеров и маркетинга:** интернет-магазины (e-commerce). Причины:
- У них максимальная концентрация нарушений: трекеры аналитики/ретаргетинга (Meta Pixel, GA4) почти всегда стоят до согласия ради конверсии;
- Есть дополнительный слой законов потребительской защиты (право на возврат, реквизиты продавца, условия оферты) — богаче на находки, нагляднее демо;
- Выше денежная мотивация владельца исправить — бизнес получает прямой доход, есть бюджет.

Решение: **не ограничиваем сканер только e-commerce**, но приоритизируем e-commerce-специфичные правила во второй волне (см. §7.3) и используем интернет-магазины как основной пример в демо/маркетинге.

## 3. Пользовательский путь (end-to-end)

1. **Главная страница** — одно поле ввода URL, кнопка «Проверить сайт». Ничего лишнего. Под полем — короткая строка: «Проверим куки, согласие на отслеживание, политику конфиденциальности и HTTPS. Бесплатно, без регистрации.»
2. **(Опционально) Уточняющие вопросы** — показываются после ввода URL, до запуска скана, свёрнуты по умолчанию под ссылкой «Уточнить для более точной оценки»: примерный размер компании (соло / малый бизнес / SME / крупная), было ли уже нарушение раньше. Если пропущены — используются консервативные значения по умолчанию.
3. **Экран сканирования** — индикатор прогресса (даже простой спиннер в v1; в v2 — пошаговый статус: «Открываю сайт… Ловлю куки… Проверяю баннер согласия… Формирую отчёт»).
4. **Отчёт**:
   - Шапка: URL, дата скана, дисклеймер (закреплён вверху, не в подвале).
   - Сводка: общее число находок по уровню серьёзности (Critical/High/Medium/Low), общий диапазон финансового риска (например «€8 000 – €45 000»), с указанием, на каком допущении о размере компании основан расчёт (и ссылкой изменить).
   - Список находок, сгруппированный по серьёзности, свёрнутые карточки. При разворачивании: понятное объяснение простым языком, техническая деталь, точная локализация «где» (скриншот с подсветкой элемента или структурированные данные — имя куки, домен, момент появления), ссылка на статью закона, диапазон штрафа с факторами, конкретные шаги по исправлению.
5. **Призыв к действию** (без давления): «Хотите переслать этот отчёт разработчику?» — кнопка «Скопировать ссылку» / «Скачать PDF» (v1.5, см. §11).

## 4. Технологический стек

Один язык, один репозиторий, один процесс запуска — максимально просто для прототипа, разрабатываемого одним человеком.

| Слой | Технология | Роль |
|---|---|---|
| Фреймворк | **Next.js (App Router) + TypeScript** | И фронтенд, и бэкенд (API routes) в одном проекте |
| Стили | **Tailwind CSS** | Быстрая, аккуратная вёрстка отчёта |
| Сканер | **Playwright (chromium)** | Открывает чужой сайт в headless-браузере, ловит cookies/сеть/DOM — критично поймать состояние *до* согласия пользователя |
| Доступность (v2) | **axe-core**, встраивается через Playwright | Проверки EAA — добавляется отдельным модулем правил, не трогая ядро движка |
| AI-объяснения | **Claude API (`claude-opus-4-8`)** | См. §5 — реализовано как переключаемый слой, не обязательная зависимость |

Запуск: `npm install && npx playwright install chromium && npm run dev`. Без очереди задач и без базы данных в v1 — Next.js работает как обычный процесс (`next start`), не как serverless-функция (у Playwright там проблемы с размером Chromium-бинарника и холодным стартом). Отчёт возвращается как JSON и рендерится на клиенте.

## 5. AI-слой: сделан переключаемым, а не обязательным

В сессии всплыли два разных ответа на вопрос «нужен ли AI» — в одной ветке «да, добавить сразу», в другой (параллельный прогон) — «по правилам, без AI». Оба ответа на деле правильные для разных сценариев, поэтому в архитектуре это решается одной настройкой, а не выбором «или-или»:

- **Режим «Только правила» (по умолчанию)** — отчёт строится чисто из базы правил: заранее написанный человеком текст объяснения и рекомендации на каждое правило (в `rules/*.ts`). Работает без API-ключа, без затрат, мгновенно, детерминированно. Это основной режим для прототипа — не требует настройки Claude API, чтобы показать сервис кому угодно.
- **Режим «AI-объяснения» (опционально включается)** — та же находка дополнительно обогащается одним батч-вызовом `claude-opus-4-8` через `output_config.format` (structured JSON output), который берёт уже готовые находки и статьи закона (без права придумывать цитаты) и переформулирует их в более естественном, персонализированном под конкретный сайт языке, плюс добавляет более конкретные шаги по исправлению с учётом деталей находки.

Технически оба режима используют один и тот же конвейер `Finding[]` → отчёт; AI — это необязательный шаг обогащения, который можно включить флагом (`AI_EXPLANATIONS_ENABLED=true` + `ANTHROPIC_API_KEY`). Если ключа нет — сервис просто не показывает переключатель и работает в режиме «Только правила».

## 7. Каталог правил

### 7.1 v1 — Cookie / GDPR / ePrivacy (уже в реализации)

| # | Правило | Улика | Закон | Базовая серьёзность |
|---|---|---|---|---|
| 1 | Куки установлены до согласия | `cookies.ts`, снимок до взаимодействия | ePrivacy Directive + GDPR ст. 6 | Critical |
| 2 | Известный трекер грузится до согласия (GA, GTM, Meta Pixel) | `network.ts` + `trackers.ts` | GDPR ст. 6, ePrivacy | Critical |
| 3 | Баннер согласия отсутствует полностью | `consentBanner.ts` | ePrivacy Directive | Critical |
| 4 | Баннер: асимметрия трения (принять — 1 клик, отклонить — много) | `consentBanner.ts` | GDPR ст. 4(11) — «свободно данное согласие» | High |
| 5 | Баннер: нет видимой кнопки отказа | `consentBanner.ts` | GDPR ст. 4(11) | High |
| 6 | Баннер: чекбоксы согласия предустановлены | `consentBanner.ts` | GDPR ст. 4(11), дело Planet49 (CJEU) | High |
| 7 | Отсутствует/недоступна ссылка на Политику конфиденциальности | `links.ts` | GDPR ст. 13 | Medium |
| 8 | Нет HTTPS / есть mixed content | `tls.ts` | GDPR ст. 32 (безопасность обработки) | Medium–High |

### 7.2 v2 — Доступность (European Accessibility Act, действует с 28.06.2025)

Встраивается через `axe-core`, прогоняемый внутри той же Playwright-страницы. **Важно** (из исследования): автоматика ловит только ~30–40% реальных нарушений доступности — отчёт обязан честно писать «автоматическая проверка покрывает часть требований EAA, не гарантирует полное соответствие», иначе продукт рискует повторить ошибку accessiBe (штраф FTC $1M + классовый иск за заявление «виджет = соответствие»).

Примеры проверок: отсутствующий `alt`, недостаточный контраст, отсутствующие `label` для полей форм, некорректная иерархия заголовков, невалидный HTML, недоступность с клавиатуры.

### 7.3 v1.5 — Специфика e-commerce (защита потребителей ЕС)

Отдельный модуль правил, добавляемый без изменения ядра движка — приоритет после v1, так как e-commerce выбран как флагманский сегмент для демонстрации:

| Правило | Улика | Закон |
|---|---|---|
| Нет реквизитов продавца (юрлицо, адрес, e-mail) в футере | текстовый поиск по DOM | Директива об электронной коммерции 2000/31/EC |
| Нет ссылки на условия использования/оферту | `links.ts` | Права потребителей ЕС |
| Нет упоминания права на возврат 14 дней | текстовый поиск | Директива 2011/83/EU |
| IAB TCF consent string невалиден / отсутствует, хотя баннер заявляет о нём | `network.ts` | GDPR ст. 6 (по прецеденту: бельгийский DPA признал IAB TCF framework незаконным в 2022) |

## 8. Модель оценки финансового риска

**Главный принцип (юридически обязательный, не опциональный): никогда не выдаётся единая точная сумма.** Всегда диапазон + список факторов, которые на него повлияли.

- **Опорные точки из реальной латвийской практики (DVI — Datu valsts inspekcija):**
  - €7 000 (2019) — малый интернет-магазин, нарушение права на удаление данных (ст. 17), отягчено несотрудничеством с регулятором.
  - €1 200 000 (2022) — SIA «TET», крупный телеком-оператор; изначальная сумма от DVI была €3 200 000, снижена на 50% за сотрудничество и меры по устранению.
- **Статутные потолки GDPR (ст. 83):** нижний уровень — до €10 млн или 2% годового мирового оборота (что больше); верхний уровень — до €20 млн или 4% (для нарушений базовых принципов обработки, прав субъектов данных, трансграничных передач — сюда попадают нарушения согласия на куки).
- **11-факторный тест (ст. 83(2))**, определяющий реальную сумму штрафа — большинство факторов физически нельзя вычислить по URL: умысел/небрежность, длительность и тяжесть, число затронутых субъектов, предыдущие нарушения, сотрудничество с надзорным органом, извлечённая финансовая выгода и т.д.
- **Практический механизм в продукте:** 1–2 лёгких уточняющих вопроса (размер компании, разовое/повторное нарушение) сужают диапазон до одного из нескольких заранее откалиброванных «уровней экспозиции» (малый бизнес / SME / крупная компания), а не пытаются высчитать точную формулу.
- **Обязательный, хорошо заметный дисклеймер** (по образцу FineMyData — лучшая практика рынка): *«Это образовательная оценка возможных последствий, а не юридическая консультация. Реальный размер штрафа определяет Datu valsts inspekcija (Латвия) или соответствующий надзорный орган ЕС индивидуально, с учётом множества факторов, которые сканирование сайта не может учесть.»*
- **Категорический запрет формулировок** вида «исправьте это — и вы соответствуете закону» / «100% compliance guaranteed». Только: «эта конкретная проблема устранена» / «риск снижен». Это прямой урок из кейса accessiBe (FTC $1M + классовый иск за именно такое обещание), подтверждённый исследованием: 25% всех исков по доступности в 2024 году были поданы именно на сайты с «инструментами соответствия».

## 9. Дизайн отчёта — UX-принципы (из анализа лучших практик рынка)

- **Сначала сводка, потом детали.** WebAIM и лучшие аудит-отчёты сходятся: нетехническому пользователю нужен «взгляд с высоты» перед списком.
- **4-уровневая шкала серьёзности** (Critical / High / Medium / Low), где уровень = воздействие на пользователя × распространённость по сайту, а не техническая сложность исправления.
- **Локализация «где» — визуальная, не только текстовая.** Для находок, привязанных к DOM-элементу (баннер) — скриншот с подсветкой рамкой конкретного элемента через `locator.screenshot()`. Это единственное, чего не делает толком ни один конкурент — главный визуальный дифференциатор продукта.
- **Простой язык сначала, техническая деталь — по клику.** Каждая карточка находки: одна понятная строка сразу видна, полное объяснение и код/скриншот — при разворачивании.
- **Конкретные шаги по исправлению**, а не только констатация проблемы — конкурент Osano выигрывает именно этим против чистых детекторов.

## 10. «Суперфичи» — roadmap после MVP

Список отсортирован примерно по ценности/сложности, не обязательство — материал для обсуждения после того, как MVP подтвердит саму идею.

1. **PDF-экспорт отчёта** — чтобы переслать разработчику/юристу без ссылки на сервис.
2. **Мониторинг во времени** — периодический повторный скан того же сайта, уведомление на e-mail, если появились новые нарушения или пропали старые.
3. **Многостраничный обход сайта** — не только главная страница, а несколько ключевых страниц (checkout, форма контактов, страница логина) — именно там чаще всего стоят трекеры.
4. **Сравнение конкурентов** — сканировать 2–3 URL параллельно, показать сравнительную таблицу рисков (полезно для агентств и B2B-продажи).
5. **API для агентств/разработчиков** — программный доступ к сканеру, чтобы веб-студии встраивали проверку в свой процесс сдачи проекта.
6. **White-label / мультиклиентский дашборд** — для агентств, ведущих десятки сайтов клиентов.
7. **Многоязычный вывод отчёта** (LV / EN / RU) — совпадает с реальной языковой средой Латвии.
8. **Персонализация оценки штрафа** — ввод реального оборота компании для более точного диапазона (вместо категорий «малый/средний/крупный»).
9. **Публичный «значок скана»** — с обязательной осторожной формулировкой («сканировано [дата], найдено N проблем», НЕ «сертифицировано соответствующим») — юридически рискованная фича, требует отдельного разбора перед реализацией.
10. **Правила под другие юрисдикции ЕС** — параметризация базы правил не только под Латвию, но и под другие страны (структура `rules/` уже это допускает).
11. **Расширение доступности (EAA) и e-commerce-правил** из §7.2 и §7.3.
12. **Browser-расширение** — быстрый скан текущей открытой вкладки без перехода на сайт сервиса.

## 11. Монетизация (не приоритет сейчас, но зафиксировано для будущего)

Возможные модели, к которым можно вернуться после валидации идеи: бесплатный разовый скан → платный PDF-экспорт / мониторинг по подписке; отдельный тариф для агентств (white-label, множественные сайты); pay-per-scan API для разработчиков. Решение по монетизации сознательно отложено — сейчас фокус на том, чтобы сам сканер работал и был полезен.

## 12. Метрики успеха прототипа

Поскольку это проверка идеи, а не готовый продукт, важно заранее определить, что считать «сработало»:
- Скан реально ловит очевидные нарушения на тестовых сайтах (GA до согласия и т.п.) без ложных срабатываний на выборке из 5–10 сайтов.
- Отчёт понятен человеку без юридического образования — он в состоянии объяснить своими словами, что не так и что делать.
- Оценка диапазона штрафа выглядит правдоподобно и не встречает возражений при показе кому-то со знанием контекста (не «$1 за нарушение» и не «всегда $20 млн»).
